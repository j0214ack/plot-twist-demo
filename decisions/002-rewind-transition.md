# 002 — 分支結束後的高速倒帶轉場

日期：2026-07-19

## 問題

分支影片播完後必須停在最後一幀，讓玩家主動按下倒帶按鈕。按下後，畫面要高速反向播放到該分支的第一幀，再顯示下一輪解讀輸入；切換期間不可插入 poster 或其他 placeholder 圖片。

這裡的「倒帶」是敘事轉場，不是提供玩家任意拖曳或編輯時間軸的影片工具。

## 既有決策

`001-screen-replacement-layer.md` 只規範影片上的螢幕替換層，沒有規範播放方向或分支結束狀態。本決策不推翻既有 ADR。

## 選項

1. 不做真正倒播，按鈕只用 CSS 閃爍或直接跳回第一幀。素材最少，但無法呈現導演規劃的「倒帶回 21:42」轉場。
2. 在瀏覽器用 `currentTime` 連續往回 seek。無須新增影片檔，但 H.264 倒向 seek 受 keyframe 間距影響，桌機與手機都可能卡頓、跳格或短暫黑畫面。
3. 每個分支預先產生一支無聲 reverse MP4，網站在使用者點擊後以 4× 正向播放該 reverse 檔。會增加影片儲存量，但瀏覽器只做最穩定的正向解碼，畫面連續且容易預載。
4. 使用 Canvas/WebCodecs 解碼並自行反向繪製影格。控制最完整，但需要額外解碼、記憶體管理與相容性處理，超出 Hackathon vertical slice。

## 共用假設

- 倒帶是一段短轉場，不需要聲音反播。
- 每個 runtime 分支最終會交付一支可獨立播放的 MP4。
- 允許影片 pipeline 在交付時多輸出一支 reverse MP4。
- 「最開始」指目前分支影片的第一幀；正式素材可把共同開場包含在分支成片開頭，讓該幀對應故事起點。

## 決定

採用選項 3。

每個故事分支新增 `rewindSrc`。原始分支播完後保留原 `<video>` 停在最後一幀，倒帶按鈕直接疊在該影格上。點擊後，把已預載的 reverse MP4 疊在原影片上，以 4×、靜音播放。reverse MP4 播完時，先把底層原影片 seek 到 0，收到 `seeked` 後才移除倒帶層並顯示下一輪輸入。

## 為什麼

這個做法把瀏覽器難以穩定完成的反向解碼移到離線影片 pipeline，runtime 只多一個明確狀態與一個可選素材欄位。它能保證按鈕出現在真正的最後一幀，且倒帶開始、結束都不必顯示 placeholder。未來若即時影片技術原生支援可靠倒播，可以保留 `rewindSrc` 作為相容 fallback。
